## 3. 两组对照实验（核心内容）

### 实验 A：IO 密集 —— 多线程通常更合适

**场景**：模拟多次「网络/磁盘等待」（`time.sleep` 代表阻塞 IO）。

| 方案 | 做法 | 预期观感 |
| --- | --- | --- |
| 串行 | 顺序 sleep 4 次，每次 0.4s | 总耗时 ≈ 1.6s |
| 多线程 | 4 个线程并行 sleep | 总耗时 ≈ 0.4s 量级 |

**要说明的点**：等待期间线程可让出，其它线程继续；这类任务多线程收益明显。

### 实验 B：CPU 密集 —— 多进程通常更合适

**场景**：模拟纯计算（循环累加 / 素数计数等短 CPU 任务）。

| 方案 | 做法 | 预期观感 |
| --- | --- | --- |
| 多线程 | 4 个线程一起算 | 耗时往往接近串行（受 GIL 限制） |
| 多进程 | 4 个进程一起算 | 耗时明显下降（能用多核） |

**要说明的点**：CPython 同一时刻通常只有一个线程执行 Python 字节码；CPU 密集应考虑进程（或换到释放 GIL 的库，如部分 numpy 操作——本示例点到为止）。

## 4. 代码结构设计（写入 yaml 的教程代码）

建议单文件、两个清晰入口，注释即教材：

```text
concurrency_demo.py（逻辑结构）
├── io_bound_task()          # 模拟 IO 等待
├── cpu_bound_task()         # 模拟 CPU 计算
├── run_io_serial()
├── run_io_threads()         # concurrent.futures.ThreadPoolExecutor
├── run_cpu_threads()
├── run_cpu_processes()      # concurrent.futures.ProcessPoolExecutor
└── if __name__ == "__main__"
        依次打印四组耗时对比
```

选用 **`concurrent.futures`** 的原因：

- 比手写 `threading.Thread` / `multiprocessing.Process` 更短、更现代
- `ThreadPoolExecutor` / `ProcessPoolExecutor` 对仗清晰，适合教学
- 强调：`ProcessPoolExecutor` 必须保护在 `if __name__ == "__main__":` 下（Windows/macOS spawn）

可选补充（注释中一笔带过，不单独开实验）：

- `threading` 模块的存在
- 进程间通信成本更高（序列化参数/返回值）
- 线程共享内存 vs 进程内存隔离

## 6. 知识便利贴（一句话摘要 · 草案）

实现阶段再定稿写入 yaml `notes`，草案如下：

1. **GIL**：CPython 有全局解释器锁，同一时刻通常只有一个线程在执行 Python 字节码。  
2. **IO 密集**：等待 IO 时可释放 GIL，多线程能重叠等待，往往更快。  
3. **CPU 密集**：纯计算难靠多线程加速，更宜多进程（或多机/原生扩展）。  
4. **ThreadPoolExecutor**：线程池，适合并发跑多个 IO 型任务。  
5. **ProcessPoolExecutor**：进程池，适合并行跑多个 CPU 型任务。  
6. **`if __name__ == "__main__"`**：多进程入口保护，避免子进程重复 import 造成递归创建。  
7. **友情提示**：线程共享内存要当心竞态；进程更安全但创建与通信成本更高。

另建 `知识便利贴-多线程与多进程.md` 展开说明（与计算器 / 分发糖果同一套文档习惯）。
