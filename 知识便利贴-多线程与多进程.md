## 示例三：IO 密集型（多线程更合适）

### 1. 多线程适合 IO 密集

IO 密集任务大部分时间在**等待**（网络、磁盘等），CPU 经常空闲。多个线程可以在等待时切换，让 CPU 去推进其它任务，整体效率通常高于纯顺序执行。

### 2. time 模块模拟 IO

本示例用 `time.sleep(...)` 假装一次阻塞等待，用来对比「顺序 / 多线程 / 多进程」的耗时差异，而不是真的发网络请求。

### 3. time.perf_counter() 测耗时

`time.perf_counter()` 适合测量**时间间隔**（单调时钟），用「结束时刻 − 开始时刻」得到一段代码跑了多久。

### 4. ThreadPoolExecutor / ProcessPoolExecutor

`concurrent.futures` 里：

- `ThreadPoolExecutor`：线程池，提交一批任务给多个线程执行  
- `ProcessPoolExecutor`：进程池，提交一批任务给多个进程执行  

接口类似，方便对照实验。

### 5. with 自动管理池

```python
with ThreadPoolExecutor(...) as pool:
    ...
```

`with` 会在离开代码块时自动关闭线程池/进程池（等待未完成任务、释放资源），不必手写繁琐的启动与关闭逻辑。

### 6. list(pool.map(...))

`pool.map(...)` 返回的是**惰性迭代器**：不消费它时，不能保证你关心的「全部任务已结束」。  
外面再包 `list(...)`，会强制把所有结果取出来，相当于「等全部任务跑完」，这样计时才完整。

## 示例四：CPU 密集型（多进程更合适）

### 1. GIL 限制多线程并行

CPython 有 **GIL（全局解释器锁）**：同一时刻通常只有一个线程在执行 Python 字节码。  
因此纯计算（CPU 密集）时，多线程往往**很难**真正吃满多核，耗时可能接近顺序执行。

### 2. 多进程适合 CPU 密集

多进程各自有解释器与内存空间，可以在多核上**并行计算**，通常比多线程更适合 CPU 密集型任务（代价是进程创建与进程间通信更重）。

### 3. ThreadPoolExecutor / ProcessPoolExecutor（对照）

CPU 示例里同样用这两种池做对照：线程池往往加速有限，进程池通常更快——用来亲眼看到 GIL 的影响。

### 4. with 与 list(pool.map(...))

与 IO 示例相同：`with` 负责自动关闭池；`list(pool.map(...))` 强制等全部任务完成后再结束计时。
