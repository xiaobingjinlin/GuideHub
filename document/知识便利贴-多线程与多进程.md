## IO 密集型（进程 / 线程 / 协程都合适）

### 1. 多线程适合 IO 密集

IO 密集任务大部分时间在**等待**（网络、磁盘等），CPU 经常空闲。多个线程可以在等待时切换，让 CPU 去推进其它任务，整体效率通常高于纯顺序执行。

### 2. 多协程适合 IO 密集

多次等待可以重叠，耗时往往接近多线程。任务很少时（如本例 4 个）和多线程几乎一样快；**通常要到几十个以上任务**，协程更轻的开销优势才会更明显。具体 `asyncio` API 见菜单「同步与异步」。

### 3. time 模块模拟等待

本示例用 sleep 假装一次阻塞等待，用来对比四种执行方式的耗时，而不是真的发网络请求。

### 4. time.perf_counter() 测耗时

`time.perf_counter()` 适合测量**时间间隔**（单调时钟），用「结束时刻 − 开始时刻」得到一段代码跑了多久。

### 5. ThreadPoolExecutor / ProcessPoolExecutor

`concurrent.futures` 里：

- `ThreadPoolExecutor`：线程池  
- `ProcessPoolExecutor`：进程池  

接口类似，方便对照实验。

### 6. with 自动管理池

```python
with ThreadPoolExecutor(...) as pool:
    ...
```

`with` 会在离开代码块时自动关闭线程池/进程池，不必手写繁琐的启动与关闭逻辑。

### 7. list(pool.map(...))

`pool.map(...)` 返回的是**惰性迭代器**。外面再包 `list(...)`，会强制把所有结果取出来，相当于「等全部任务跑完」，这样计时才完整。

## CPU 密集型（多进程更合适）

### 1. GIL 限制多线程并行

CPython 有 **GIL（全局解释器锁）**：同一时刻通常只有一个线程在执行 Python 字节码。  
因此纯计算（CPU 密集）时，多线程往往**很难**真正吃满多核，耗时可能接近顺序执行。

### 2. 多进程适合 CPU 密集

多进程各自有解释器与内存空间，可以在多核上**并行计算**，通常比多线程更适合 CPU 密集型任务。

### 3. 多协程不擅长 CPU 密集

若协程内部没有让出，事件循环会被这段计算堵住，实际近乎串行——用来对照「协程擅长等，不擅长算」。asyncio 细节仍见「同步与异步」。

### 4. ThreadPoolExecutor / ProcessPoolExecutor（对照）

CPU 示例里用线程池与进程池对照：线程池往往加速有限，进程池通常更快——用来亲眼看到 GIL 的影响。

### 5. with 与 list(pool.map(...))

与 IO 示例相同：`with` 负责自动关闭池；`list(pool.map(...))` 强制等全部任务完成后再结束计时。
