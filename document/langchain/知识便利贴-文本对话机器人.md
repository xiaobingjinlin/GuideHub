# 知识便利贴 · 文本对话机器人

本文对应示例：**LangChain → 文本对话机器人**。右侧「知识便利贴」只展示每条的一句话摘要；下文为展开说明。

核心链路（LangChain 1.3，不用 LangGraph / 不用 RunnableWithMessageHistory 热路径）：

```text
SqliteChatMessageHistory
  → ChatPromptTemplate 组消息
  → trim_messages（字数护栏）
  → ChatQwen.stream（思考 + 回答）
  → add_messages 落库
  → finalize_after_turn（混合压缩）
```

---

## 1. SqliteChatMessageHistory 与 SQLite

自研 `SqliteChatMessageHistory`（实现 `BaseChatMessageHistory`），按 `session_id` 写入本地 SQLite；模块级复用连接，并开 WAL。不再依赖已日落的 `langchain-community`。

## 2. ChatPromptTemplate 组上下文

用 `system + MessagesPlaceholder("history") + human` 组装本轮 prompt，与落库历史同源，避免「模板一套、流式又手写一套」。

## 3. ChatQwen（langchain-qwq）与思考流式

通用 `ChatOpenAI.stream` **不会**提取通义扩展字段 `reasoning_content`。  
本示例对 Qwen3 使用 `langchain-qwq.ChatQwen`：`enable_thinking=True` 时，流式 chunk 把思考放在 `additional_kwargs["reasoning_content"]`，正式回答在 `content`。

（`ChatQwQ` 面向 `qwq-*` / `qvq-*`；本菜单模型是 `qwen3.7-plus`，用 `ChatQwen`。）

## 4. trim_messages 护栏

送模型前按字符预算裁剪（保留 system、尽量从 Human 边界起留尾部）。  
**轮数窗口**仍由 `chat_compress` 混合摘要负责；`trim_messages` 只防单次请求过长。

## 5. stream + yield：先思考、后回答

`iter_chain_stream` 对每个 chunk：先 `yield ("reasoning", …)`，再 `yield ("content", …)`。  
成功后 `add_messages([Human, AI])` 一次写入（思考过程不入库，只展示在左侧气泡）。

## 6. 一轮对话的收尾流程

```text
发送 → iter_chain_stream → finalize_after_turn
                              ├─ 空回复兜底
                              ├─ 满 20 轮混合压缩
                              └─ 刷新右侧「对话历史」
```

点「终止」不落库；左侧可保留气泡，刷新后以数据库为准。
